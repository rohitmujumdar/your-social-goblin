The memory layer — the part judges score

Two requirements, met by one idea: nothing lives in the app — everything lives in HydraDB and is recalled.

persistent memory logic

It recalls past sessions on its own

Every coin, profile, and repair is a memory in HydraDB. The dashboard holds no state — it is rebuilt from a live recall on every load.

  • Restart the server (a new session) → the whole world reassembles from memory.
  • A relationship you repaired stays repaired, recalled from its resolution memory.
  • The trace panel shows every autonomous WRITE / QUERY as it happens.
context-aware execution

Its advice changes with what it remembers

Before suggesting a fix it recalls who the person is and what worked last time, then tailors the message.

no memory"Send a follow-up."
with recalled memory"Send a short honest line — long excuses don't land with Marc, and short honesty worked last time."
  • Marc → short honesty · Sarah → a concrete date. Same app, different advice.
  • Repair outcomes are stored as rules and recalled into future suggestions.
write a pattern → persist in HydraDB → recall next session → adapt the recommendation → write the outcome → repeat

The layers, top to bottom

A small office. Only the brain talks to the AI, and only the clerk talks to HydraDB. Clean doors.

Sources & adapters
seed.py · (future parsers)
The front-ends. Turn any chat (seeded today; WhatsApp / iMessage / paste tomorrow) into one normalized shape: {sender, day, text}.
Front desk · API
main.py (FastAPI)
The counter. The endpoints the frontend calls: /ingest /dashboard /coin /repair /trace.
The brain
agent.py → Nebius LLM
Interprets language. Finds the dark pattern in a chat, and writes the personalized repair message. Falls back to a safe answer key if the model misbehaves.
The memory clerk
hydra.py
The only door to HydraDB. Writes notes, reads notes, and logs every single read/write to the trace panel.
HydraDB — the notebook
memories: coins · profiles · repairs
The memory of record. Everything that persists lives here. Survives restarts — this is what makes it remember across sessions.
The scorekeeper
patterns.py
Pure math, no AI. Turns "how many open coins" into a health % and a Goblin size. Deterministic, so the demo is fast and repeatable.

Two paths through the stack

Writing a memory and reading it back are the only two things that ever happen. Everything in the game is one of these.
